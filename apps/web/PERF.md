# Замеры скорости сайта (Lighthouse CI)

OSS-инструмент для мониторинга, насколько быстро работает фронт. Гоняет Google Lighthouse
по продакшн-сборке и печатает оценки + Core Web Vitals. Запускать перед релизом или когда
правил что-то «тяжёлое» (картинки, шрифты, скрипты) — чтобы поймать регресс скорости.

## Как запустить

```bash
pnpm --filter web perf
```

Что происходит: `next build` → поднимает `next start` на :3000 → прогоняет Lighthouse по
3 раза на каждой странице (берётся медиана) → печатает оценки и пишет HTML-отчёты в
`apps/web/.lighthouseci/` (gitignored, открыть в браузере).

> Нужен установленный Chrome (Lighthouse им рулит). Порт 3000 должен быть свободен.

## Что меряем

Страницы: `/` (лендинг) и `/pricing` — публичные, на скорость которых смотрят первыми.

Метрики (категории 0–100 + сырые Core Web Vitals):
- **Performance** — общая оценка скорости.
- **LCP** (Largest Contentful Paint) — когда отрисовался главный контент. Цель < 2.5s.
- **CLS** (Cumulative Layout Shift) — дёрганье вёрстки при загрузке. Цель < 0.1.
- **TBT** (Total Blocking Time) — сколько main-thread заблокирован JS. Цель < 300ms.
- плюс Accessibility / Best-practices / SEO.

Пороги — в `lighthouserc.json` (сейчас `warn`: не валят команду, только предупреждают).
Чтобы сделать жёстким гейтом — поменять `"warn"` → `"error"` для нужной метрики.

## Где смотреть результат

- В консоли — сводка оценок и какие ассершены сработали.
- `apps/web/.lighthouseci/` — полные HTML-отчёты по каждому прогону (детальный разбор:
  что тормозит, что грузится, водопад запросов).

## Дальше (опционально)

- Гонять в CI (GitHub Actions) на каждый PR → история скорости во времени.
- Реальные пользователи (field / RUM): Next умеет `useReportWebVitals` без новых зависимостей —
  отдаёт LCP/INP/CLS живых юзеров. Lighthouse выше — это лабораторный (синтетический) замер.
