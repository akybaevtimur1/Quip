## 1. С чего начинать: ДИЗАЙН-ПЕРВЫМ (но вертикально). Почему так.

Обычная лучшая практика для продакшн-запуска соло-фаундера с уже работающим ядром:
**сначала зафиксировать дизайн-направление и систему, потом строить экраны в ней, потом
вшивать логику — поэкранно, а НЕ «сначала все макеты, потом весь код».**

Почему дизайн ведёт:
1. **Переделка дизайна — самая дорогая.** Зафиксируешь язык (палитра/тип/компоненты/motion)
   один раз → всё остальное плагинится без переписывания.
2. **UI — рычаг привлечения** (приоритет фаундера: «захватывать кликами, видом»). Лендинг и
   первый экран решают, останется ли юзер. Это дизайн-задача.
3. **Логика ядра УЖЕ работает.** Новая логика (auth/оплата) — тонкая и механическая
   (Supabase-схема и billing уже написаны в T6) → она легко вшивается в готовые экраны.

⚠️ Ловушка, которой избегать: НЕ делать «красивые мёртвые макеты». Строить РЕАЛЬНЫЕ
компоненты (Next 16 + дизайн-система), чтобы логика втыкалась прямо в них.

### Порядок фаз (top = первым)
- **D1 — Дизайн-направление (ЗАМОК).** Изучить референсы (`design-md/` + quip.ink) → собрать
  конкретную дизайн-систему (палитра, типографика, spacing, компоненты, motion) + 2-3 концепта
  лендинга. ПОКАЗАТЬ фаундеру (он не дизайнер — нужны картинки/скрины), он выбирает направление.
- **D2 — Лендинг.** Маркетинговый сайт в принятой системе (уровень quip.ink): hero, демо/как
  работает, отличитель (объяснимый хук + reframe без флешей), прайсинг-секция, CTA, соц-док.
  Захват: микро-анимации, живое демо-превью, чёткий первый клик.
- **A1 — Авторизация.** Экраны вход/регистрация/сессия в системе + Supabase Auth (`@supabase/ssr`,
  схема из T6 готова, профиль авто-создаётся триггером). Гейт на приложение.
- **D3 — Дашборд / оболочка приложения.** Логин-зона: список проектов/клипов, интеграция
  существующего редактора, аккаунт, индикатор расхода/лимита. Когерентно дизайн-системе.
- **P1 — Прайсинг вживую + оплата.** Страница тарифов + Lemon Squeezy (вебхук → profiles.plan) +
  гейт квоты в create_job (402, `check_quota` готов) + usage на Postgres.
- **POLISH — designer's-eye проход** (design-review/тайт-проверка) по всем экранам.

Если фаундер захочет логику раньше — можно, но дизайн-систему всё равно зафиксировать в D1.

## 2. ДИЗАЙН-МАНДАТ (это критично — отнестись максимально серьёзно)

### 2.1 Обязательные источники стиля (ЧИТАТЬ перед дизайном)
- **`design-md/`** — ~70 разобранных дизайн-систем брендов (каждый = `DESIGN.md` с палитрой/
  типографикой/motion). **ОБЯЗАТЕЛЬНО изучить релевантные** под наш домен (AI-видео-инструмент,
  премиум-крафт): рекомендую **runwayml, elevenlabs, linear.app, vercel, cursor, raycast, framer,
  superhuman** (+ при желании stripe/notion/posthog). Вытащить из них: тёмная премиум-подача,
  hairline-борды, измеренная типографика с негативным трекингом, один-два акцента, продукт-
  скриншоты в тёмных панелях, сдержанный motion.
- **quip.ink** (лендинг-референс фаундера) — north-star тона/верстки. Свериться вживую (browse/
  defuddle/WebFetch), понять, что в нём цепляет, не копировать буквально.
- Текущая палитра ядра — **Warm Charcoal + Coral** (`apps/web/app/globals.css` `@theme`).
  Можно ЭВОЛЮЦИОНИРОВАТЬ в продакшн-уровень (не обязательно сохранять 1:1), но решение — через
  концепты, показанные фаундеру.

### 2.2 Скиллы дизайна — использовать ПО МАКСИМУМУ (как просил фаундер)
Подключать релевантные на каждой фазе (это огромная часть ценности):
- **design-consultation** — старт D1: понять продукт, исследовать ландшафт, предложить полную
  систему + превью шрифтов/цветов.
- **imagegen-frontend-web** — концепты лендинга (ОДНА горизонтальная картинка на КАЖДУЮ секцию).
- **design-shotgun** — несколько вариантов дизайна → борд сравнения → структурный фидбек.
- **brandkit** — айдентика/логотип-система/бренд-борд, если нужен бренд.
- **frontend-design** + **ui-ux-pro-max** — продакшн-реализация (распознавать и ИЗБЕГАТЬ
  «AI-slop»/дженерик; 50+ стилей, палитры, пары шрифтов, UX-гайды).
- **taste-skill** (анти-слоп) — пре-флайт чек, чтобы не выглядело шаблонно.
- **emil-design-eng** — полировка микро-деталей и анимаций (то, что делает софт «дорогим»).
- **design-review** / **ios-design-review-аналог** — designer's-eye QA в конце.
Скиллы можно добавлять/убирать под задачу. Цель — НЕ дженерик, цепляющий вид: клики,
микро-motion, ясная иерархия, быстрый «второй взгляд».

### 2.3 Дизайн НЕ хромает и КОГЕРЕНТЕН
Новый UI единым языком с дизайн-системой D1; токены в `globals.css` `@theme`; паттерны
`components/*`. Показывать фаундеру скрины (browse/скриншоты) — он судит глазами.

## 4. Правила автономной сессии

- **OSS-first** (память `feedback-oss-first`): перед своим кодом — искать готовое (auth-паттерны
  `@supabase/ssr`, UI-примитивы shadcn/ui, лендинг-паттерны). Сверяться с АКТУАЛЬНОЙ докой
  (context7/официальные): Next 16, @supabase/ssr, Lemon Squeezy, Tailwind v4.
- **Использовать агентов** для fan-out (изучение нескольких design-md разом, параллельные экраны).
- **Дизайн показывать скринами** (это не-дизайнер-фаундер); ключевые развилки — спросить
  (AskUserQuestion) с превью-вариантами, НЕ угадывать визуальный вкус.
- **Ядро/редактор НЕ ломать** — оборачивать. Не трогать `services/captions-revideo-spike/`, репо лендинга.
- `just check` зелёный + `next build` зелёный перед коммитом; conventional commits; типы из `models.py`
  (`just types`); тест-первым на pure-логике. **Перезапускать воркер** после правок его кода.
- PowerShell: PATH-refresh, абсолютные пути, коммит `-F` файл.
- Аккаунты/секреты/деплой — фаундер (Supabase URL/anon/service_role, Lemon, домены, перевыпуск кук).
- Новая ветка от HEAD (напр. `feat/production-shell`). Документировать (CLAUDE.md журнал + HANDOFF).
