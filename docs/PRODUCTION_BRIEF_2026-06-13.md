## 1. С чего начинать: ДИЗАЙН-ПЕРВЫМ (но вертикально). Почему так.

Обычная лучшая практика для продакшн-запуска соло-фаундера с уже работающим ядром:
**сначала зафиксировать дизайн-направление и систему, потом строить экраны в ней, потом
вшивать логику — поэкранно, а НЕ «сначала все макеты, потом весь код».**

Почему дизайн ведёт:
1. **Переделка дизайна — самая дорогая.** Зафиксируешь язык (палитра/тип/компоненты/motion)
   один раз → всё остальное плагинится без переписывания.
2. **UI — рычаг привлечения** (приоритет фаундера: «захватывать кликами, видом»). Лендинг и
   первый экран решают, останется ли юзер. Это дизайн-задача.
3. **Логика ядра УЖЕ работает.** Новая логика (auth/оплата) — тонкая и механическая
   (Supabase-схема и billing уже написаны в T6) → она легко вшивается в готовые экраны.

⚠️ Ловушка, которой избегать: НЕ делать «красивые мёртвые макеты». Строить РЕАЛЬНЫЕ
компоненты (Next 16 + дизайн-система), чтобы логика втыкалась прямо в них.

### Порядок фаз (top = первым)
- **D1 — Дизайн-направление (ЗАМОК).** Изучить референсы (`design-md/` + quip.ink) → собрать
  конкретную дизайн-систему (палитра, типографика, spacing, компоненты, motion) + 2-3 концепта
  лендинга. ПОКАЗАТЬ фаундеру (он не дизайнер — нужны картинки/скрины), он выбирает направление.
- **D2 — Лендинг.** Маркетинговый сайт в принятой системе (уровень quip.ink): hero, демо/как
  работает, отличитель (объяснимый хук + reframe без флешей), прайсинг-секция, CTA, соц-док.
  Захват: микро-анимации, живое демо-превью, чёткий первый клик.
- **A1 — Авторизация.** Экраны вход/регистрация/сессия в системе + Supabase Auth (`@supabase/ssr`,
  схема из T6 готова, профиль авто-создаётся триггером). Гейт на приложение.
- **D3 — Дашборд / оболочка приложения.** Логин-зона: список проектов/клипов, интеграция
  существующего редактора, аккаунт, индикатор расхода/лимита. Когерентно дизайн-системе.
- **P1 — Прайсинг вживую + оплата.** Страница тарифов + Lemon Squeezy (вебхук → profiles.plan) +
  гейт квоты в create_job (402, `check_quota` готов) + usage на Postgres.
- **POLISH — designer's-eye проход** (design-review/тайт-проверка) по всем экранам.

Если фаундер захочет логику раньше — можно, но дизайн-систему всё равно зафиксировать в D1.

## 1.5 ПОНЯТЬ ПРОДУКТ ГЛУБОКО (перед дизайном — обязательно)

Нельзя дизайнить/писать копирайт «вообще». Сначала внутренне понять, ЧТО мы продаём и КОМУ:

- **Что это:** длинное видео (подкаст/интервью/стрим/лекция) → 3–10 вертикальных клипов с
  прожжёнными субтитрами, цепляющим хуком сверху и объяснением, ПОЧЕМУ момент сработает.
- **Кому:** соло-креаторы, подкастеры, коучи, эксперты, маркетологи — кто репёрпозит длинный
  контент в шортсы и НЕ хочет вручную пересматривать часы.
- **Наш wedge (отличитель, на нём строить весь нарратив):** НЕ «30+ клипов в клик» (это Vizard,
  максимизируют output, размылись в комбайн), а **«меньше клипов, но ты понимаешь, ПОЧЕМУ их
  стоит постить»** — максимизируем confidence. Конкретно: объяснимость (hook + why_works + score
  + type на клип) + reframe без флешей (Δ=0) + сочные субтитры + честный прайсинг без казино.
- **Боли конкурентов (бить в них копирайтом):** AI мажет в выборе моментов; мутные кредиты +
  surprise-paywall; overbuilt-комбайн. Мы — ясность, честность, фокус.
- **«Aha»-момент (к нему вести воронку как можно быстрее):** юзер видит свои 3–6 клипов С
  ОБЪЯСНЕНИЕМ почему каждый сработает + гладкий вертикальный кадр. Это надо показать в первые
  минуты (живое демо на лендинге + быстрый первый прогон после регистрации).

**Перед дизайном ОБЯЗАТЕЛЬНО** прочитать ядро (apps/web/app/page.tsx, components/editor/*,
services/worker/app/models.py, run.py) — чтобы строить ИМЕННО под этот продукт, а не дженерик-SaaS.

## 2. ДИЗАЙН-МАНДАТ (это критично — отнестись максимально серьёзно)

### 2.1 Обязательные источники стиля (ЧИТАТЬ перед дизайном)
- **`design-md/`** — ~70 разобранных дизайн-систем брендов (каждый = `DESIGN.md` с палитрой/
  типографикой/motion). **ОБЯЗАТЕЛЬНО изучить релевантные** под наш домен (AI-видео-инструмент,
  премиум-крафт): рекомендую **runwayml, elevenlabs, linear.app, vercel, cursor, raycast, framer,
  superhuman** (+ при желании stripe/notion/posthog). Вытащить из них: тёмная премиум-подача,
  hairline-борды, измеренная типографика с негативным трекингом, один-два акцента, продукт-
  скриншоты в тёмных панелях, сдержанный motion.
- **quip.ink** (лендинг-референс фаундера) — north-star тона/верстки. Свериться вживую (browse/
  defuddle/WebFetch), понять, что в нём цепляет, не копировать буквально.
- Текущая палитра ядра — **Warm Charcoal + Coral** (`apps/web/app/globals.css` `@theme`).
  Можно ЭВОЛЮЦИОНИРОВАТЬ в продакшн-уровень (не обязательно сохранять 1:1), но решение — через
  концепты, показанные фаундеру.

### 2.2 Скиллы дизайна — использовать ПО МАКСИМУМУ (как просил фаундер)
Подключать релевантные на каждой фазе (это огромная часть ценности):
- **design-consultation** — старт D1: понять продукт, исследовать ландшафт, предложить полную
  систему + превью шрифтов/цветов.
- **imagegen-frontend-web** — концепты лендинга (ОДНА горизонтальная картинка на КАЖДУЮ секцию).
- **design-shotgun** — несколько вариантов дизайна → борд сравнения → структурный фидбек.
- **brandkit** — айдентика/логотип-система/бренд-борд, если нужен бренд.
- **frontend-design** + **ui-ux-pro-max** — продакшн-реализация (распознавать и ИЗБЕГАТЬ
  «AI-slop»/дженерик; 50+ стилей, палитры, пары шрифтов, UX-гайды).
- **taste-skill** (анти-слоп) — пре-флайт чек, чтобы не выглядело шаблонно.
- **emil-design-eng** — полировка микро-деталей и анимаций (то, что делает софт «дорогим»).
- **design-review** — designer's-eye QA в конце (несогласованности, слоп, медленные интеракции).
- **chrome-devtools / lighthouse / browse** — ОБЯЗАТЕЛЬНО для верификации: реальные скрины
  экранов фаундеру + Lighthouse (perf/a11y/SEO/best-practices, цель 90+) + проверка интеракций.
- **supabase** — для auth/схемы/RLS (актуальные конвенции, security-чеклист).
ИСПОЛЬЗОВАНИЕ СКИЛЛОВ — НЕ ОПЦИОНАЛЬНО: они дают правильные принципы и анти-слоп. На КАЖДОЙ
фазе подключать релевантные. Цель — НЕ дженерик, цепляющий тактильный вид: клики, микро-motion,
ясная иерархия, быстрый «второй взгляд».

### 2.3 Дизайн НЕ хромает и КОГЕРЕНТЕН
Новый UI единым языком с дизайн-системой D1; токены в `globals.css` `@theme`; паттерны
`components/*`. Показывать фаундеру скрины (browse/скриншоты) — он судит глазами.

### 2.4 Тактильность, цвет, анти-слоп (это «дорогой» вид)
- **Тактильность/кликабельность:** каждый интерактив имеет состояния hover/active/focus/disabled;
  кнопки «нажимаются» (press-feedback, лёгкий transform/тень); переходы 150–250ms ease;
  loading-состояния (скелетоны, не спиннеры-заглушки); пустые состояния продуманы. Motion
  ГИДИРУЕТ внимание (reveal на скролле, фокус на CTA), не декоративный хаос.
- **Цвет — дисциплина (НИКАКИХ тупых дефолтов):** палитра выводится из референсов design-md,
  1–2 акцента максимум, остальное — нейтральная шкала поверхностей/hairline. ЗАПРЕЩЕНО:
  дженерик фиолетово-синий градиент «как у всех AI», радуга, чистый #000/#FFF без тёплого
  сдвига, случайные акценты. Контраст текста ≥ WCAG AA (это и a11y, и SEO).
- **Анти-AI-slop (taste-skill ОБЯЗАТЕЛЬНО):** не центрировать всё подряд, не лепить
  стеклянные карточки везде, не одинаковые секции «иконка+заголовок+текст ×6». Асимметрия,
  иерархия, второй-взгляд-моменты, как у Linear/Vercel/Runway.

### 2.5 SEO — сделать ОТЛИЧНО (не для галочки)
- **Технический SEO (Next 16 App Router):** `generateMetadata` per-page (title/description,
  canonical), Open Graph + Twitter cards (с OG-картинкой), `app/sitemap.ts` + `app/robots.ts`,
  JSON-LD структурные данные (`SoftwareApplication`, `Organization`, `FAQPage`), правильная
  иерархия h1→h2→h3, alt на изображениях, семантический HTML (`<main>`, `<nav>`, `<section>`).
- **Производительность = SEO (Core Web Vitals):** LCP < 2.5s, CLS ~0, хороший INP. `next/image`
  (никаких голых `<img>` на лендинге), оптимизация шрифтов (`next/font`, без layout shift),
  ленивые тяжёлые блоки, минимум клиентского JS на лендинге (RSC где можно).
- **Контент-SEO:** копирайт под поисковый интент («AI video clipper», «repurpose long videos
  into shorts», «podcast to shorts», «explainable AI clips») — в h1/h2 естественно, не спам.
- **Проверка:** прогнать Lighthouse / chrome-devtools (perf + a11y + SEO + best-practices),
  целиться в 90+; приложить вывод в DoD.

### 2.6 Конверсия / воронка / привлечение (бизнес-модель в дизайне)
- **Воронка:** Лендинг → регистрация → ПЕРВЫЙ КЛИП (активация = «aha») → апгрейд. Минимизировать
  трение до первой ценности (как можно быстрее показать объяснимые клипы).
- **Лендинг конвертит:** ясный value-prop выше сгиба (мессадж «Don't just get clips. Know why
  they're worth posting.»), ЖИВОЕ демо/превью результата, отличитель (объяснимый хук + reframe
  без флешей), один основной CTA, прозрачный прайсинг (free показывает, что даёт — анти-surprise-
  paywall), соц-док/доверие. Тон — уверенный, не хайповый.
- **Активация/онбординг:** после регистрации довести до первого клипа за минимум шагов (вставил
  ссылку → получил объяснимые клипы). Честный free-preview с watermark.
- **Этичная персуазия (это часть нашего бренда):** НИКАКИХ дарк-паттернов, фейк-таймеров,
  скрытых списаний — это ровно то, за что ругают конкурентов; наша честность = преимущество.

## 4. Правила автономной сессии

### 4.0 Качество кода (НИКАКОГО говнокода)
- Архитектура: переиспользуемые дизайн-система-примитивы (Button/Input/Card/Dialog/…), НЕ
  god-компоненты, НЕ инлайн-магия/хардкод-цвета (только токены `@theme`). Типизировано (TS strict),
  без `any`. Доступность (a11y: роли, фокус, клавиатура, контраст) — это и SEO, и качество.
- Бэкенд по железным правилам CLAUDE.md: pure-функции под тестами, TDD-первым на логике,
  без тихих фолбэков (JobError), типы из models.py (just types).
- Гейт: `just check` зелёный + `next build` зелёный + Lighthouse 90+ перед коммитом фазы.
- Структура папок осмысленная (app/(marketing), app/(app), app/(auth) route-groups; components/ui,
  components/marketing, components/app). Server Components по умолчанию, client — где нужна интерактивность.

- **OSS-first** (память `feedback-oss-first`): перед своим кодом — искать готовое (auth-паттерны
  `@supabase/ssr`, UI-примитивы shadcn/ui, лендинг-паттерны). Сверяться с АКТУАЛЬНОЙ докой
  (context7/официальные): Next 16, @supabase/ssr, Lemon Squeezy, Tailwind v4.
- **Использовать агентов** для fan-out (изучение нескольких design-md разом, параллельные экраны).
- **Дизайн показывать скринами** (это не-дизайнер-фаундер); ключевые развилки — спросить
  (AskUserQuestion) с превью-вариантами, НЕ угадывать визуальный вкус.
- **Ядро/редактор НЕ ломать** — оборачивать. Не трогать `services/captions-revideo-spike/`, репо лендинга.
- `just check` зелёный + `next build` зелёный перед коммитом; conventional commits; типы из `models.py`
  (`just types`); тест-первым на pure-логике. **Перезапускать воркер** после правок его кода.
- PowerShell: PATH-refresh, абсолютные пути, коммит `-F` файл.
- Аккаунты/секреты/деплой — фаундер (Supabase URL/anon/service_role, Lemon, домены, перевыпуск кук).
- Новая ветка от HEAD (напр. `feat/production-shell`). Документировать (CLAUDE.md журнал + HANDOFF).
