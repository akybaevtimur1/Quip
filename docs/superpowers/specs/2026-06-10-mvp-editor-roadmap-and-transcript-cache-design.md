# MVP Editor Roadmap + Sub-project 0 (Transcript Cache) — Design

> Дата: 2026-06-10. Статус: утверждённый дизайн (brainstorming).
> Контекст-доки: `docs/HANDOFF.md`, `CLIPFLOW_DEV_PLAN.md`.

## 1. Контекст и цель

Phase 0 (сквозной пайплайн + минимальный web + reframe v3 с Δ=0 флешей) готов.
Фаундер определил следующую цель **не** как «задеплоить продукт», а как **прочный
локальный/демо-инструмент с настоящим редактором поверх результата ИИ** — уровня
TikTok / CapCut / Opus. Это то, что план называл «Phase 2 — Магия», вынесенное в MVP.

Облако, очередь (K1), R2, auth, оплата — **вне scope** этого MVP (Phase 1+).

## 2. End-state: как MVP выглядит в итоге

Локальный стек (web :3000 + worker :8000). Пользователь:

1. Вставляет YouTube-ссылку или загружает файл → пайплайн (download → transcribe →
   Gemini выбор моментов → reframe 9:16 → captions → render) выдаёт N клипов-кандидатов.
2. Открывает любой клип в **редакторе**, где:
   - видит видео 9:16 с субтитрами **поверх видео в реальном времени**;
   - **двигает границы шортса ручками** на таймлайне (а не вводом секунд);
   - **редактирует текст субтитров прямо на видео** (не в отдельном боксе);
   - **выбирает стиль субтитров** из пресетов;
   - субтитры **анимированы в стиле TikTok** — пословное появление, подсветка
     отдельных слов, лёгкие анимации;
   - может включить/выключить **двойные субтитры** (опция-тумблер).
3. Экспортирует финальный MP4 (попиксельно совпадающий с превью).

Экономика: повторные прогоны одного видео **не платят** за транскрипцию (кэш);
исследуется удешевление транскрипции через Gemini.

> Рекомендованный чекпоинт перед показом реальным пользователям (не блокер MVP):
> прогнать формальный golden-set eval (§7.4 плана) — убедиться, что качество нарезки
> Q ≥ 60% и reason_agrees ≥ 70%. Фаундер сейчас судит качество глазами; формальный
> прогон стоит сделать до того, как звать внешних людей.

## 3. Декомпозиция (утверждённый порядок)

Каждый под-проект = отдельный цикл spec → plan → реализация.

| # | Под-проект | Что входит | Зависит от |
|---|-----------|-----------|-----------|
| **0** | **Transcript cache** (этот документ) | Content-addressed кэш транскрипции + ограничение размера. Быстрый выигрыш, делаем первым. | — |
| **1** | **Caption Engine v2** | Архитектура превью (browser-render vs server-render) + TikTok-стили (word-pop, подсветка, анимации) + пресеты + экспорт, совпадающий с превью + тумблер двойных субтитров. | 0 |
| **2** | **Visual Editor UI** | Таймлайн с draggable-trim, редактирование текста субтитров на видео, выбор стиля, живое превью. Поверх backend `app/editor/`. | 1 |

### Ключевое открытое решение для Под-проекта 1 (решается в его брейншторме)

**Как рендерится превью в редакторе.** Сейчас субтитры = ASS, прожжённый ffmpeg
(≈секунды на рендер) — для живого редактора с анимациями это слишком медленно.
Кандидаты: **(a) браузерный рендер** (Remotion или Canvas/DOM поверх `<video>`) с
финальным экспортом, попиксельно совпадающим с превью; **(b) серверный ре-рендер**
на каждую правку (просто, но медленно). Рекомендация по умолчанию — (a) на базе
Remotion (план это и предвидел), но финальное решение — в брейншторме Под-проекта 1.

## 4. Детальный дизайн — Под-проект 0: Transcript cache

### Проблема

`POST /jobs` (`main.py:65`) генерит новый случайный `job_id` → новая папка
`data/<job_id>/` → Stage 1 всегда зовёт Deepgram. Существующий кэш в `run.py:95`
срабатывает только по наличию `transcript.json` в **той же** папке (повтор того же
`job_id`). Новый UI-джоб того же видео платит за Deepgram снова.

Доминанта стоимости — Deepgram (~$0.14/33мин против ~$0.016 Gemini). Скачивание
бесплатно (только время). Транскрипт зависит **только от аудио + провайдера/модели**.

### Решение: двухуровневый кэш

- **Уровень 1 (есть):** `transcript.json` в папке джоба — повтор того же `job_id`.
- **Уровень 2 (новый):** общий контент-кэш `data/_cache/transcripts/`, ключ =
  `hash(source.wav) + provider + model`. Повтор любого ранее транскрибированного
  аудио (даже под новым `job_id`) → Deepgram не вызывается, cost=0.

Поток в Stage 1 (`run.py`), downstream без изменений (по-прежнему читает
`transcript.json` из папки джоба):

```
если transcript.json в папке джоба есть       → берём его (уровень 1, как сейчас)
иначе:
   key = cache_key(audio_sha(source.wav), provider, model)
   если data/_cache/transcripts/<key>.json есть → копируем в transcript.json, cost=0
   иначе                                         → Deepgram → пишем И в transcript.json,
                                                     И в кэш; затем evict(...)
```

### Новый модуль `app/transcript_cache.py`

Границы как в проекте (pure-логика изолирована и покрыта тестами; I/O — тонкие обёртки):

- `cache_key(audio_sha: str, provider: str, model: str) -> str` — **PURE**, тест ПЕРВЫМ
  (правило №3). Включает provider+model → смена модели = другой ключ = нет устаревших
  отдач.
- `audio_sha(wav: Path) -> str` — sha256 байтов (тонкий I/O).
- `get_cached(cache_dir: Path, key: str) -> Transcript | None` — чтение + Pydantic-валидация.
- `put_cached(cache_dir: Path, key: str, transcript: Transcript) -> None` — запись.
- `select_evictions(entries, now, *, max_entries, max_age_sec) -> list[str]` — **PURE**,
  тест ПЕРВЫМ. Возвращает имена под удаление: сначала старше `max_age_sec` (TTL), затем,
  если всё ещё > `max_entries`, самые старые по mtime до попадания в лимит.
- `evict(cache_dir, *, max_entries, max_age_sec) -> None` — собирает (имя, mtime),
  зовёт `select_evictions`, удаляет.

### Конфиг (`config.py`, с дефолтами)

| Переменная | Дефолт | Описание |
|------------|--------|----------|
| `TRANSCRIPT_CACHE_ENABLED` | `true` | Выключатель кэша. |
| `TRANSCRIPT_CACHE_MAX_ENTRIES` | `200` | Кап по числу файлов (LRU по mtime). |
| `TRANSCRIPT_CACHE_MAX_AGE_DAYS` | `60` | TTL записи (дни). |

Кэш-папка `data/_cache/` уже покрыта gitignore (`data/`).

### Что НЕ делаем (YAGNI / backlog)

- **Не пропускаем download** на YouTube cache-hit — это экономит время, не деньги
  (отдельная оптимизация: ключ по video-id, позже).
- Не делаем распределённый/сетевой кэш — локальный диск.

### DoD

1. `cache_key` и `select_evictions` — unit-тесты написаны ПЕРВЫМИ и зелёные.
2. Прогон одного видео под `job_A`, затем того же под `job_B`: во втором прогоне в логе
   `transcribe: cached (hash hit)`, `transcribe_cost=0`, в `runs.jsonl` транскрипция = $0.
3. `evict` срабатывает: после превышения cap/TTL старые файлы удалены (проверка тестом
   на временной директории).
4. `just check` зелёный (lint + mypy + tsc + test-unit + anti-drift) перед коммитом.
