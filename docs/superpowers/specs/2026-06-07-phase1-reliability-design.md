# ClipFlow Phase 1 — «Usable MVP: надёжность» — дизайн

> Статус: дизайн одобрен фаундером 2026-06-07. Следующий шаг — implementation-план (writing-plans).
> Источник Phase 0: `CLIPFLOW_DEV_PLAN.md` (§5 чеклист, §9 roadmap). Дисциплина §4А действует.

## TL;DR

Phase 0 доказал, что тонкий пайплайн режет клипы насквозь (A→J, реальный прогон через UI).
Phase 1 делает из него **надёжный продукт без «магии»**: настоящая очередь с ретраями,
авто-язык, кэш транскрипции, progressive-вывод клипов, изоляция хранилища, 1–2 стиля
субтитров, дашборд экономики. **Локально-первым.** Деплой — отдельный под-этап на
ОТДЕЛЬНОМ хостинге (лендинг не трогаем). «Магия» (trim-редактор, why-panel) — Phase 2.

## Зафиксированные решения (не релитигируем)

| Решение | Выбор | Почему |
|---|---|---|
| Очередь | **RQ (Redis Queue) + Redis** | проще Celery, sync-воркеры подходят (ffmpeg/httpx блокирующие), ретраи из коробки; задачи изолированы за интерфейсом — Celery подставится позже при нужде |
| Параллелизм | несколько задач/юзеров одновременно | воркер-пул RQ |
| Язык | **авто-определение** (Deepgram `detect_language`) | у фаундера RU-контент; EN-only давал 0 клипов |
| Деплой | **отдельный под-этап, отдельный хостинг** (Vercel `clipflow-web` + Railway + Redis + R2) | правило №10: лендинг `Shorts-Automatizator` не трогаем ВООБЩЕ |
| Хранилище | local-fs сейчас, за интерфейсом `storage.py` | R2 — в деплой-под-этапе, свап одним файлом |
| БД | SQLite (WAL) остаётся | Postgres — Phase 3 (auth); для локального параллелизма WAL+очередь хватает |
| Прогресс | polling (как в Phase 0) | SSE/WS — Phase 2; не усложняем |

## Не-цели Phase 1 (YAGNI — сознательно НЕ делаем)

- ❌ Auth / аккаунты / оплата (Phase 3).
- ❌ SSE/WebSocket (Phase 2; polling достаточно).
- ❌ R2 и реальный деплой В САМОМ Phase 1 (отдельный под-этап после).
- ❌ «Магия»: transcript-trim редактор, «why these» панель, fly-out (Phase 2).
- ❌ Postgres, Modal/GPU, self-host WhisperX (позже).
- ❌ Per-frame панорамирование/диаризация 2+ спикеров (Phase 4).

## Архитектура

```
Браузер (Next)
   │  POST /jobs                         GET /jobs/{id} (polling 2.5s)
   ▼                                         ▲
FastAPI (main.py) ── enqueue ──► Redis ──► RQ worker (отдельный процесс)
   │  insert queued                              │  tasks.run_pipeline_job
   ▼                                             ▼  (ретраи/таймауты)
SQLite (db.py, WAL) ◄──── статус + клипы по мере готовности ──── run_pipeline (Stage 0→5)
   ▲                                             │
   └── storage.py (put/get/url; local сейчас) ◄──┘  артефакты в data/<job_id>/
```

- **FastAPI** только принимает/отдаёт (enqueue + чтение SQLite). Тяжёлую работу не делает.
- **RQ worker** — отдельный процесс (`just worker-queue`), берёт задачи из Redis, гоняет
  `run_pipeline_job`. Падение задачи → ретрай по политике; исчерпан → статус `failed`.
- **run_pipeline** (из Phase 0) переиспользуется как есть; добавляем хук progressive-вывода
  и кэш транскрипции. Стадии остаются pure + file-based (идемпотентны).
- **Границы** (как в Phase 0): `pipeline/*` чистые; склейка в `run.py`/`tasks.py`;
  контракты — `models.py` → codegen. Никаких тихих фолбэков → `JobError`/failed.

## Компоненты (K1–K7)

### K1. Очередь + надёжность (костяк)
- `app/queue.py`: подключение к Redis, объект очереди RQ. `app/config.py`: `REDIS_URL`.
- `POST /jobs`: `db.insert_job(queued)` + `queue.enqueue(run_pipeline_job, job_id, ...)`.
- Воркер-процесс: `rq worker` (рецепт `just worker-queue`).
- **Ретраи с бэк-оффом** (pure-расчёт расписания, TDD): транзиентные сбои (сеть, 429/503,
  yt-dlp) → N ретраев с экспоненциальным бэк-оффом; не-транзиентные (битый вход) → сразу failed.
- **Таймаут на стадию**: каждая стадия имеет потолок времени → JobError при превышении (R5).
- Статусы в SQLite на границах стадий (как сейчас, через `on_status`).
- **DoD:** 5 задач разом обрабатываются; транзиентный сбой ретраится и проходит; убил воркер
  посреди задачи → после рестарта задача до-выполняется или помечается failed (не «висит»).

### K2. Кэш транскрипции по `hash(source)` (R6)
- Контент-адресный кэш: ключ = `sha256(source.wav)` (или source.mp4). Транскрипт кладём в
  `data/_cache/transcripts/<hash>.json`. Перед Deepgram — проверка кэша.
- **DoD:** 2-й прогон того же видео → `transcription cost $0`, транскрипт взят из кэша
  (в `runs.jsonl` видно `cached`). Главный рычаг экономии при итерации промпта D.

### K3. Авто-определение языка
- `stage1_transcribe`: убрать захардкоженный `language="en"`; передать Deepgram
  `detect_language=true` (или `language=None`); `transcript.language` = `detected_language`.
- Промпт выбора моментов остаётся англоязычной ИНСТРУКЦИЕЙ, но транскрипт подаётся на родном
  языке; `reason` модель вернёт осмысленно. Субтитры — на родном.
- **DoD:** RU-видео → `transcript.language="ru"`, слов > 0, клипы есть.

### K4. Progressive output
- `run_pipeline` дописывает каждый готовый `ClipOut` в SQLite сразу после рендера (а не
  только в конце); `db` хранит частичный `clips_json`; `GET /jobs/{id}` отдаёт частичный список.
- Фронт (`useJob`) уже polling-ит; `ClipGrid` показывает то, что пришло (клипы появляются
  по одному). Pure-логика мерджа/сортировки — под тест.
- **DoD:** во время прогона `GET` показывает растущее число клипов; UI рисует их постепенно,
  не дожидаясь конца.

### K5. Storage-абстракция (`put/get/url`)
- `app/storage.py`: интерфейс `put(local_path, key) -> None`, `get(key) -> bytes/path`,
  `url(key) -> str`. Реализация `LocalStorage` (сейчас: файлы в `data/`, url = `/media/...`).
- Воркер/`main.py` обращаются к файлам ТОЛЬКО через storage. R2-реализация — заглушка +
  ADR, чтобы свап в деплой-под-этапе был один файл.
- **DoD:** медиа отдаётся через storage-интерфейс; `LocalStorage` покрыт; R2-стаб + ADR.

### K6. Стили субтитров (1–2 brand-neutral)
- `stage4_captions`: вынести стиль в именованные пресеты (напр. `bold-center`,
  `bold-bottom`), параметр `caption_style` в конфиге/контракте; `build_ass` принимает стиль.
- **DoD:** рендер со стилем A и B → оба валидный ASS, визуальный спот-чек.

### K7. Cost/latency дашборд
- Чтение `runs.jsonl` → агрегатор (pure: mean/p90 cost/TTFC/total_sec по прогонам) →
  отдать на `GET /metrics` (JSON) + простая страница в web (`/metrics`) или CLI-сводка.
- **DoD:** дашборд показывает строки прогонов + сводку mean/p90 (юнит-экономика на виду).

## Контракты данных (изменения)

- `models.py`: возможно добавить `caption_style` в конфиг-уровень (не в wire, если UI не выбирает);
  `Job`/`ClipOut` — без изменений (clip↔source linkage + `words[]` уже есть с Phase 0).
- Менять контракты ТОЛЬКО через `models.py` → `just types` (anti-drift гейт).

## Обработка ошибок

- Транзиентные (сеть/429/503/yt-dlp) → ретраи с бэк-оффом (K1). Исчерпаны → `failed` + текст.
- Не-транзиентные (битый вход, источник > лимита, 0 слов) → сразу `failed`/empty, без ретраев.
- Таймаут стадии → `JobError(stage, "timeout")` → failed. Никаких тихих фолбэков.

## Тестирование (TDD-цели, pure)

- Расписание бэк-оффа ретраев (детерминированно).
- Ключ кэша транскрипции (hash стабилен, одинаковый вход → одинаковый ключ).
- Классификация ошибки транзиентная/нет.
- Progressive-мердж/сортировка клипов в статусе.
- Агрегация дашборда (mean/p90) на фикстуре `runs.jsonl`.
- Билдер стиля субтитров (валидный ASS для каждого пресета).
- Контракт-тесты на изменённый ответ Deepgram (detect_language).
- Сеть/Redis — мок/фикстуры; реальный прогон — `just e2e`.

## Деплой (отдельный под-этап, ПОСЛЕ Phase 1, отдельный хостинг)

- Vercel: **новый проект** `clipflow-web` (root `apps/web`). Лендинг-проект не трогаем.
- Railway: воркер (FastAPI + RQ worker + Redis addon) + persistent volume или R2 для файлов.
- R2: реализация `storage.py` (presigned url), lifecycle (клипы/transcript храним, upload-source ~60 дней, YouTube-import не храним).
- Секреты только на хостинге; `.env` в git не уходит.

## Порядок и зависимости

1. **K1** (костяк очереди) — первым, остальное на нём.
2. **K2, K3** — независимы, можно параллельно (кэш + язык).
3. **K4** (progressive) — после K1.
4. **K5** (storage) — перед деплой-под-этапом; полезно и локально.
5. **K6, K7** — независимы (стили, дашборд).
6. Деплой-под-этап — последним, отдельно.

## Зависимость от GO-gate

Phase 1 строим в предположении, что качество нарезки приемлемо. По плану §7.4 формальный
GO (Q≥60%, reason_agrees≥70%) — оценка фаундера на golden set. Если на итерациях качество
не вытянет — приоритет смещается на «винтик» (промпт/модель/ICP), а не на инфру Phase 1.
