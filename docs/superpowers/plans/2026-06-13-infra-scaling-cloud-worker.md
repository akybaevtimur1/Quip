# Инфра-масштабирование: cloud worker + auth + очередь — ДИЗАЙН для следующей сессии

> **Статус:** дизайн/хендофф. НЕ исполнялось. Написано в сессии с уже большим контекстом,
> чтобы следующая сессия начала с чистого листа и сделала это как отдельный большой проект.
> **Решение фаундера (2026-06-13):** целимся в РЕАЛЬНЫЙ масштаб (100+ одновременных юзеров),
> воркер **уезжает в облако** (не локальная машина + туннель).
>
> **Этот документ заменяет «деплой» часть** старого auth-плана
> (`2026-06-13-auth-analytics-plan.md` Task 13 + ngrok + SQLite-only). Auth-логика оттуда
> (Supabase, JWT, /admin) ОСТАЁТСЯ валидной — меняется ГДЕ и КАК это крутится.

---

## 0. TL;DR — что выяснили и что меняем

Фаундер спросил: «если 100 человек одновременно зайдут — не будет ли тормозить?». Разбор
показал, что вопрос распадается на ДВА разных:

1. **Зайти на сайт / залогиниться / листать** → масштабируется без проблем. Это Vercel (CDN,
   автоскейл) + Supabase (автоскейл, 50k MAU, безлимит API). 100 человек открывают страницу —
   никто не ждёт.
2. **Запустить обработку видео** → ВОТ ТУТ стена. Каждый джоб + каждый поллинг статуса идёт на
   ОДНУ домашнюю машину через туннель. Это и есть бутылочное горло.

**Корневые проблемы текущей архитектуры (local worker + free ngrok):**

| Проблема | Факт | Следствие |
|---|---|---|
| free ngrok (с фев 2026) | 20k запросов/мес, 1 ГБ/мес, сессия **2 часа**, случайный URL | Поллинг (раз в 2.5с = 24 req/min/юзер): **100 юзеров выжрут месячный лимит за ~8 минут**. URL меняется каждые 2ч → надо вручную править env на Vercel. Скачивание клипов через туннель убивает 1 ГБ. |
| Один процесс FastAPI | Тяжёлые джобы в `BackgroundTasks` делят 40-слотовый threadpool с sync-эндпоинтами | Несколько ffmpeg/ASD-джобов (~2× длины клипа на CPU) **затыкают пул → поллинг и auth-проверки встают в очередь** за ними. Это и есть «у кого-то грузится вечность». |
| Нет очереди / лимита параллелизма | 100 джобов стартуют разом на одной машине | CPU-thrash, всё ползёт, возможен OOM. Одна домашняя CPU физически не транскодит 100 видео параллельно. |
| SQLite без WAL/busy_timeout | конкурентные записи статусов | `database is locked` → тихие падения джобов. |
| `/media` локально | файл лежит на той машине, что рендерила | при нескольких воркерах клип недоступен с другого инстанса; bandwidth через туннель. |

**Вывод:** auth-слой написан правильно, но «домашний воркер + туннель» не выдержит реальную
нагрузку. Нужен облачный воркер с очередью, общим стейтом и объектным хранилищем.

## 1. Целевая архитектура (для масштаба)

```
            ┌─────────────────────────────────────────────┐
            │  VERCEL (Next.js)  — автоскейл, CDN          │
            │  • Supabase Auth (getClaims = локальная     │
            │    верификация JWT, без сетевого хита)       │
            │  • Читает СВОИ джобы напрямую из Supabase    │
            │    Postgres (RLS) + Realtime-подписка        │
            │    (НЕ поллит воркер!)                       │
            └───────────────┬──────────────┬──────────────┘
        создать джоб /      │              │  статус (push)
        правки (REST)       ▼              ▼
   ┌────────────────────────────┐   ┌──────────────────────────┐
   │  API (FastAPI), stateless  │   │  SUPABASE (облако)        │
   │  • несколько реплик        │   │  • Auth (users)           │
   │  • НЕ делает тяжёлую работу │──▶│  • Postgres: jobs (RLS    │
   │  • кладёт джоб в очередь    │   │    по user_id) + Realtime │
   └──────────────┬─────────────┘   │  • Storage (клипы) / или  │
                  │ enqueue          │    Cloudflare R2 + CDN    │
                  ▼                  └──────────────────────────┘
        ┌───────────────────┐
        │  REDIS (очередь)   │
        └─────────┬─────────┘
                  │ consume (bounded concurrency)
        ┌─────────▼──────────────────────────┐
        │  WORKER(ы) — тяжёлые, CPU/GPU       │
        │  • RQ/Celery consumer               │
        │  • 1–2 джоба на бокс, скейл боксами │
        │  • пайплайн stage0–5 (без изменений)│
        │  • рендер → грузит mp4 в Storage    │
        │  • пишет статус/cost в Postgres     │
        └────────────────────────────────────┘
```

Ключевая идея: **тяжёлая работа НИКОГДА не в request-пути**, **стейт и медиа общие**
(не на диске воркера), **клиент не поллит воркер** (Realtime), **юзеры изолированы на уровне БД**
(RLS).

## 2. Решения по компонентам (с обоснованием)

### 2.1 Стейт джобов: SQLite → Supabase Postgres (RLS по user_id)
- **Почему:** несколько API + worker инстансов не могут делить один SQLite-файл. Нужна общая БД.
  Supabase Postgres уже оплачен (free 500 МБ), джобы — крошечные строки.
- **RLS:** политика `user_id = auth.uid()` изолирует юзеров на уровне БД (defense-in-depth).
  Фронт читает СВОИ джобы напрямую (без round-trip к воркеру). Воркер пишет через
  `service_role` (минует RLS).
- **Перенос:** логика `db.row_to_wire` портируется почти как есть; колонка `user_id` (уже в
  auth-плане) становится FK на `auth.users`. `clip_edits` тоже в Postgres.
- ⚠️ Это убирает почти всю «J1 SQLite»-обвязку — закладывать время.

### 2.2 Поллинг → Supabase Realtime
- **Почему:** поллинг раз в 2.5с — ДОМИНАНТНЫЙ источник запросов и то, что плавит туннель и
  threadpool. Если строки джобов в Postgres — клиент подписывается на `jobs:id=eq.<jobId>` и
  получает push при апдейте. Воркер пишет статус → клиент мгновенно видит. **Воркер вообще
  перестаёт получать запросы статуса.**
- Free: 200 одновременных realtime-коннектов, 2M сообщений/мес → 100 юзеров = 100 коннектов, ок.
- Это самое дешёвое одно изменение, снимающее «100 юзеров долбят воркер».

### 2.3 Воркер: split на API (лёгкий) + Worker (тяжёлый) + Redis-очередь
- **API:** FastAPI, stateless — только создаёт джоб (enqueue) + edit-эндпоинты (лёгкие). Несколько
  реплик, автоскейл. Тяжёлого в процессе НЕТ → нет starvation threadpool.
- **Worker:** отдельные процессы, читают очередь, **bounded concurrency** (1–2 тяжёлых джоба на
  бокс; масштаб = добавить боксы). RQ или Celery + Redis.
- В `CLAUDE.md` уже есть отложенный дизайн **K1 RQ-queue**
  (`docs/superpowers/specs/2026-06-07-phase1-reliability-design.md` +
  `.../plans/...k1-queue.md`) — переиспользовать как старт.

### 2.4 Медиа: локальный `/media` → объектное хранилище + CDN
- **Почему:** локальный файл не работает между инстансами + bandwidth через туннель/инстанс.
- **Куда:** Supabase Storage (просто, уже в экосистеме) ИЛИ Cloudflare R2 (дёшево, без egress).
  Воркер по завершении рендера грузит mp4 → отдаёт CDN-URL в `jobs`-строке. Воркеры stateless.

### 2.5 Auth at scale: асимметричные ключи JWT
- **Сейчас (auth-план):** HS256 + общий `SUPABASE_JWT_SECRET`. Работает, НО фронтовый
  `getUser()` в middleware бьёт по сети Supabase на каждый запрос (для нашего паттерна
  не критично — поллинг и так не через middleware, но это «неправильный дефолт»).
- **На масштаб:** включить **asymmetric signing keys** в Supabase →
  - фронт: `getClaims()` верифицирует JWT **локально** (без сетевого хита);
  - воркер: верифицирует **RS256/ES256 через JWKS** (`PyJWT` + `PyJWKClient`, кэш ключей),
    НЕ через HS256-секрет.
- ⚠️ Coupling: если включить асимметрию, `app/auth.py` из auth-плана (`algorithms=["HS256"]`,
  shared secret) **сломается** — переписать на JWKS. Если оставить HS256 — фронт `getClaims`
  локально НЕ верифицирует (нет публичного ключа у клиента для HS), будет фолбэк в сеть.
  **Решение:** на масштаб — асимметрия везде.

### 2.6 Платформа (куда деплоить)
| Вариант | Плюс | Минус |
|---|---|---|
| **Fly.io Machines** | scale-to-zero, per-second биллинг, хорош для бёрстовых тяжёлых джобов | кривая обучения |
| **Render** | просто, есть «background workers» из коробки | дороже на тяжёлом CPU |
| **Railway** | очень просто | дорого на масштабе |
| **Hetzner (Docker)** | лучший $/ядро для ffmpeg, дёшево | сам админишь |
| **GCP Cloud Run** | отличен для лёгкого API | request cap 60 мин, нет персистентного воркера → только под API |
- **Рекомендация:** API на PaaS (Fly/Render) + тяжёлые воркеры на дешёвом CPU (Hetzner) или
  Fly Machines, скейлящиеся по глубине очереди. GPU позже (NVENC для энкода, GPU-ASD) если
  латентность/стоимость потребуют.
- **Стоимость:** доминанта = Deepgram транскрипция (~$0.0043/мин). Компьют вторичен → скейл
  компьюта = про латентность/параллелизм, не про деньги. Деньги = про Deepgram → лимиты на юзера.

### 2.7 Backpressure и лимиты (обязательно для публичного запуска)
- **На юзера:** макс. одновременных джобов (1–2), дневной cap джобов (контроль стоимости),
  rate-limit на создание.
- **Глобально:** cap глубины очереди; полна → «очередь занята, попробуйте позже».
- **UI:** позиция в очереди + ETA (на масштабе джобы ЖДУТ — это норм, главное показать).
- **Анти-абьюз стоимости:** чужак может кинуть 90-мин видео = дорогой Deepgram. Нужны квоты на
  длину источника по тарифу + /admin-дашборд (уже в auth-плане) для слежки за тратами.

## 4. Карта багов из auth-плана, которые НАДО учесть (carried-forward)

Эти найдены в этой сессии — пофиксить при исполнении, независимо от облака:

1. **CORS wildcard сломан.** В auth-плане Task 5: `allow_origins=["https://*.vercel.app"]` —
   Starlette НЕ поддерживает wildcard-сабдомены в `allow_origins` (сравнивает строкой буквально)
   → задеплоенный Vercel-сайт получит **CORS-блок от воркера**. Надо:
   ```python
   app.add_middleware(
       CORSMiddleware,
       allow_origins=["http://localhost:3000"],
       allow_origin_regex=r"https://.*\.vercel\.app$",   # + кастомный домен когда будет
       allow_methods=["*"],
       allow_headers=["*"],
   )
   ```
2. **SQLite WAL** (если короткий период ещё на SQLite до переезда в Postgres):
   `PRAGMA journal_mode=WAL; PRAGMA busy_timeout=5000;` в `_conn()`.
3. **`app/auth.py` HS256 → JWKS** при включении асимметричных ключей (см. 2.5).
4. **jobs-таблица → Postgres + RLS** заменяет «user_id колонку в SQLite» из auth-плана Task 4
   на масштабе.
5. **Медиа → Storage** вместо `/media` StaticFiles.
6. **middleware:** `getUser()` → `getClaims()` + React `cache()` (меньше латентности).

## 5. Фазовый роадмап (для следующей сессии разбить на под-планы)

### Фаза A — Фундамент (deploy + auth, без очереди)
Цель: рабочий облачный воркер + auth + общий стейт/медиа. Один инстанс воркера держит умеренный
параллелизм.
- A1. Dockerize воркера (API + пайплайн в одном образе; ffmpeg, torch CPU-колёса, mediapipe,
  scenedetect). ⚠️ образ тяжёлый (torch) — multi-stage, кэш слоёв.
- A2. Supabase: включить асимметричные signing keys; создать `jobs`/`clip_edits` таблицы в
  Postgres + RLS-политики (`user_id = auth.uid()`).
- A3. Воркер: переключить `db.py` с SQLite на Postgres (psycopg/asyncpg или supabase-py через
  service_role). Порт `row_to_wire`.
- A4. Медиа: рендер → upload в Storage/R2 → CDN-URL в строку джоба.
- A5. Auth: `app/auth.py` на JWKS (RS256); фронт `getClaims()`; CORS regex (см. §4).
- A6. Деплой API на Fly/Render; env-vars; healthcheck; стабильный публичный URL (без туннеля).
- A7. /admin аналитика (из auth-плана) — поверх Postgres-агрегатов.
- A8. Per-user квоты (макс. длина источника, дневной cap).

### Фаза B — Расцепление (очередь + Realtime)
- B1. Redis + RQ/Celery (переиспользовать K1-дизайн).
- B2. API только enqueue; отдельный worker-процесс consume с bounded concurrency.
- B3. Фронт: Realtime-подписка на свою jobs-строку вместо поллинга воркера.
- B4. UI позиции в очереди + ETA; backpressure при полной очереди.

### Фаза C — Скейл-аут
- C1. Автоскейл воркеров по глубине очереди (Fly Machines / Hetzner-пул).
- C2. GPU: NVENC-энкод + GPU-ASD если латентность/стоимость требуют.
- C3. Метрики/обсервабилити (cost/latency на стадию → дашборд; runs.jsonl → реальный сбор).
- C4. Глобальные лимиты, алерты на трату Deepgram.

## 6. Открытые вопросы для следующей сессии
- Storage: Supabase Storage (просто) vs Cloudflare R2 (дёшево, без egress) — выбрать по бюджету.
- Postgres-доступ из воркера: supabase-py (service_role) vs прямой psycopg к Supabase-Postgres
  (connection pooling! Supabase даёт pgBouncer-порт 6543 для serverless/множества коннектов).
- Очередь: RQ (проще) vs Celery (мощнее, сложнее) — RQ скорее всего достаточно.
- Нужен ли GPU сразу или CPU-боксы с горизонтальным скейлом дешевле на старте (вероятно CPU).
- Тариф Supabase: free-проект **засыпает после 1 недели без активности** + лимит 2 активных
  проекта → для прод-демо может понадобиться Pro ($25). Учесть.
- Лимиты на длину/кол-во видео на бесплатного юзера (анти-абьюз Deepgram-стоимости).
