# ⛔ НЕ ТРОГАТЬ — Инвариант кадровой сетки reframe (фикс флешей)

> Здесь зафиксирован инвариант, который убирает «флеши» (мелькающий кадр на переходе
> между режимами fill↔fit). Любое изменение, нарушающее этот инвариант, ВЕРНЁТ флеши —
> и заметишь ты это **только на видео ≠ 25 fps** (на 25fps-видео баг невидим, см. ниже,
> поэтому тесты могут быть зелёными, а у пользователя — флеши).

## TL;DR (что нельзя ломать)

**Геометрическая сетка reframe (склейки → шоты → границы регионов) ОБЯЗАНА быть в
НАТИВНОМ fps источника — В ТОЙ ЖЕ сетке, в которой `render_clip` режет видео.**

Конкретно в `app/pipeline/stage3_reframe.py` → `reframe_segment`:

```python
aligned_start = round(start * fps) / fps          # тот же origin, что в render_clip
total_frames  = round((end - aligned_start) * fps)# fps = НАТИВНЫЙ fps источника (meta.fps)
cuts   = detect_scene_cuts(video, aligned_start, end, fps, ...)   # ← fps, НЕ face_fps
shots  = build_shots_frames(cuts, total_frames)
tracks_native = score_tracks_in_segment(video, ..., aligned_start, end, face_fps, ...)  # ASD @25
tracks = [resample_track(t, face_fps, fps) for t in tracks_native]  # 25fps → нативный
regions = plan_regions(shots, tracks, fps, ...)    # ← fps, НЕ face_fps
```

Три вещи, которые НЕЛЬЗЯ менять, не понимая последствий:

1. **`detect_scene_cuts`, `build_shots_frames`, `plan_regions` получают `fps` (нативный),
   НЕ `face_fps`.** Если вернуть `face_fps` — границы регионов снова окажутся в сетке 25fps,
   а рендер режет в нативной → флеш.
2. **`aligned_start = round(start * fps) / fps`** — это origin клипа. `render_clip` вычисляет
   ТО ЖЕ САМОЕ из `seg_start`. Они обязаны совпадать, иначе t0 регионов уезжает на <1 кадр
   от часов filtergraph → флеш. detect/score получают `aligned_start`, НЕ сырой `start`.
3. **ASD (`score_tracks_in_segment`) остаётся на `face_fps=25`** — модель LR-ASD это требует
   (см. ниже). Поэтому дорожки лиц ОБЯЗАТЕЛЬНО прогоняются через `resample_track(t, 25, fps)`
   перед `plan_regions`. Убрать ресемпл = сетки снова разъедутся.

## Как теперь работают переходы (fill ↔ fit)

1. **Склейки источника** ищет `detect_scene_cuts` (PySceneDetect) на пересжатом в **нативный
   fps** клипе → отдаёт номера кадров склеек, frame-accurate в нативной сетке.
2. **Шоты** (`build_shots_frames`) — интервалы `(f0, f1)` между склейками, в нативных кадрах.
3. **Режим на шот** (`plan_regions`): один режим на весь шот —
   - `fit` (весь кадр + блюр-рамки), если лиц нет ИЛИ 2+ разнесённых лица (широкий план);
   - `fill` (вертикальный кроп 9:16 на говорящем), иначе. Говорящий = макс. ASD speak-score;
     при молчании ASD — фолбэк на крупнейшее лицо.
4. **Граница региона = граница шота = реальный кадр склейки.** Режим меняется ТОЛЬКО на
   склейке. Внутри шота кадр держится стабильно (никаких смен режима в середине плана).
5. **Короткие шоты** (< `reframe_min_hold_sec`, дефолт 1.5с) поглощаются предыдущим
   (`merge_short_regions`) — гасит рапид-монтаж.
6. **Рендер** (`render_clip` → `build_smooth_filter`): `[0:v]split` → на каждый регион
   `trim=start_frame=round(t0*fps):end_frame=round(t1*fps)` + crop/fit → `concat` (жёсткий
   cut). Поскольку `t0 = cut_frame / fps`, то `round(t0*fps) = cut_frame` ТОЧНО → trim режет
   ровно на склейке → **нет кадра нового шота со старым кропом → флеша физически нет.**

Жёсткий cut на границе невидим: контент в этот кадр и так прыгает (реальная склейка).
Кроссфейд/xfade между fill и fit НЕ используется намеренно — он сам читался как зум-вспышка.

## Корень бага (что было сломано)

`render_clip` всегда резал в **нативном fps** источника (`meta.fps`: 23.976 / 24 / 29.97 / 60…).
А `reframe_segment` строил склейки/шоты/регионы в **жёстко зашитой сетке `face_fps = 25`**.

Граница региона была `t0 = cut_frame_25 / 25`. Рендер вычислял кадр trim как
`round(t0 * нативный_fps)`. На 29.97 fps:

```
t0 = 16.72с (= кадр 418 в сетке 25fps)
render trim = round(16.72 * 29.97) = round(501.16) = 501   ← склейка между 501 и 502
t0 = 36.52с  →  round(36.52 * 29.97) = round(1094.5) = ...  ← РОВНО посередине кадров
```

→ trim промахивался мимо реальной склейки на **±1 кадр** → один кадр нового шота
показывался со старым кропом = **флеш**.

**Почему это не ловилось тестами / почему казалось «то работало, то нет»:**
DoD-проверка («Δ = 0 кадров, флеш невозможен») гонялась на `comedy01` и `dod01`.
**Оба ровно 25.0 fps.** На 25fps сетка планирования (25) совпадает с сеткой рендера (25),
ошибка = 0, тест зелёный — но это ничего не доказывало для реальных видео. На любом
видео ≠ 25 fps баг был всегда. Отсюда иллюзия «когда-то работало» (то было 25fps-видео).

## Почему ASD нельзя просто перевести на нативный fps

`app/asd/scorer.py` → `score_track` ЖЁСТКО зашивает 25fps:
```python
af = python_speech_features.mfcc(...)        # аудио-фичи @100 Гц
vf = vf[: int(round(length * 25)), ...]      # видео @25fps
... vf[i*dur*25 : ...]  /  af[i*dur*100 : ...]   # ratio 4:1 (25 видео : 100 аудио)
```
Модель LR-ASD обучена на 25fps видео + 100 Гц аудио (4:1). Подашь 29.97fps кадры — модель
сопоставит их с аудио как будто это 25fps → рассинхрон → мусорные speak-score → reframe
начнёт наводиться не на того человека. ПОЭТОМУ ASD остаётся на 25, а выход (дорожки лиц)
ресемплится в нативную сетку через `resample_track`.

## Что МОЖНО менять безопасно

- Пороги/кнобы: `reframe_scene_threshold`, `reframe_min_hold_sec`, `reframe_speak_threshold`,
  `reframe_smoothing`, `reframe_wide_ratio`, `reframe_speaker_crop_scale` — это тюнинг качества,
  они не трогают кадровую сетку.
- Логику выбора режима в `plan_regions` (fill/fit, кого считать говорящим) — пока границы
  регионов остаются на кадрах склеек (выход `build_shots_frames`).
- **Стартовое значение cx-траектории fill-региона** (2026-06-13, `ebfc3dc`): `plan_regions` тянет
  `prev_fill_end_cx` и инитит EMA нового fill-региона на нём (`_track_trajectory(init_cx=…)`) →
  центр едет НЕПРЕРЫВНО поперёк подряд идущих fill-шотов (нет «телепорта» камеры). Это меняет ТОЛЬКО
  стартовый cx внутри fill-региона — границы режима fill/fit и кадровая сетка НЕ тронуты → флеши
  невозможны. Сброс `prev_fill_end_cx=None` на fit/split (реальная смена плана).
- **cx-выражение рендера `build_fill_crop_expr` можно сделать piecewise-LINEAR** (сейчас piecewise-
  CONST = ступенька, ~3% шаги; превью редактора уже линейное+CSS-glide, `b571b55`). Линейный expr =
  по-настоящему плавный пан в скачанном файле. Безопасно: это cx ВНУТРИ fill-региона, границы (trim-
  кадры) не трогаются. Follow-up по запросу фаундера.
- **Ручной split региона по override (`apply_overrides_to_regions`, 2026-06-23):** override с
  произвольным под-диапазоном РЕЖЕТ покрытый регион на под-интервалы (interval-painting) и красит
  каждый по последнему покрывающему override. КАЖДАЯ новая граница ОБЯЗАНА быть frame-snapped
  (`_snap(t,fps)=round(t*fps)/fps`) в НАТИВНОМ fps; внешние края = собственные (уже on-grid) края шота.
  Не snap'нул новую границу = вернул флеш на ≠25fps. Контракт `CropOverride` не менялся (фронт шлёт
  произвольный source-диапазон, бэк снапит и режет).

## Что НЕЛЬЗЯ менять без перечитывания этого файла

- Передавать `face_fps` вместо `fps` в `detect_scene_cuts` / `build_shots_frames` / `plan_regions`.
- Убирать `resample_track` или `aligned_start` из `reframe_segment`.
- Менять `*25` / `*100` в `score_track` (это контракт модели, не «магические числа»).
- Менять формулу `aligned_start = round(seg_start * fps) / fps` в `render_clip` так, чтобы она
  разошлась с `reframe_segment`.

## Известное (статус)

- **Editor-путь МУЛЬТИ-интервал — ЗАКРЫТО (2026-06-23, свип №2).** Раньше: `render_timeline` при >1
  интервала шёл в `flatten_timeline`+`build_timeline_filter` (полный вход, БЕЗ `-ss`) и якорил
  trim-кадры на СЫРОЙ `iv.source_start` + `round(.,3)`, тогда как регионы построены `reframe_segment`
  от `aligned_start = round(source_start*fps)/fps`. На off-grid старте (нормально после трима/нарезки)
  `round((source_start+t0)*fps)` промахивал на ±1 кадр (численно 1.39% комбо на ≠25fps, 0% после) →
  флеш. Продуктовый editor-путь = `resolve_regions_accurate` → `reframe_segment` (НЕ legacy
  `resolve_regions`/`detect_cuts`-в-секундах — тот benchmark-only). Фикс: `flatten_timeline` якорит
  на тот же `aligned` → `src_f0 = round((aligned + r.t0)*fps)` = абсолютный кадр одно-интервального
  `render_clip`, Δ=0. DoD: `tmp/verify_editor_flash_multiinterval.py` (29.97fps, 2 интервала, MAX = 0.0).
- **Короткие шоты больше не «съедаются» молча (2026-06-23):** `min_scene_len` от нативного fps
  (`reframe_min_scene_sec=0.25`) вместо дефолта либы 15 кадров + `reframe_min_hold_sec` 1.5→0.8. После
  фикса A смена fill↔fit на реальной склейке = невидимый hard-cut, поэтому агрессивный merge больше не
  нужен как анти-флеш (лишь лёгкий анти-строб). Границы остаются на нативных кадрах.
- Реальная склейка, пропущенная порогом `reframe_scene_threshold`, — это «держим старый кадр на новом
  контенте», НЕ флеш (лечится порогом, не сеткой). Если нужного шота всё же нет — ручной **«Split here»**
  в редакторе ставит границу (frame-snapped) и тайтит под-диапазон.

## Как нашли (для истории — метод, а не угадывание)

1. Не угадывали. Запустили systematic-debugging.
2. Сняли подозреваемых: коммит удаления ASS — аддитивный (`subtitles`→`null`), цепочку concat
   не трогал; коммит ретраев Gemini — только `stage2_select.py`, рендера не касается.
3. Прочитали ОБА пути кадров: `detect_scene_cuts` (пересжатие `-r face_fps`, сетка 25) vs
   `render_clip` (нативный `meta.fps`). Увидели рассинхрон сеток.
4. Подтвердили данными: `comedy01`/`dod01` (фикстуры DoD) = ровно 25.0 fps; реальные видео =
   29.97/23.976/24/60. Арифметика: `36.52*29.97 = 1094.5` — ровно посередине кадров.
5. Проверили на реальном 29.97fps: после фикса MAX ошибка границ = 0.00000 кадра.

Урок: **«тест зелёный» ≠ «работает», если фикстура случайно удовлетворяет сломанному
допущению.** Для frame-accurate вещей фикстуры обязаны покрывать разные fps (24/25/29.97/60).
